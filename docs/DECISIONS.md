## ADR-001: FastAPI как основной HTTP-фреймворк
**Решение:** FastAPI.  
**Обоснование:** Async-поддержка, Pydantic-валидация, OpenAPI документация.

## ADR-002: Redis как очередь задач
**Решение:** Redis + `rq`.  
**Обоснование:** Минимальный overhead, встроенный retry с backoff, persistence (AOF).

## ADR-003: Adapter Pattern + Open-Closed Principle
**Решение:** `BaseAdapter` ABC + `AdapterRegistry`.  
**Обоснование:** Возможность добавлять новые сервисы без изменения ядра.

## ADR-004: Config-driven Generic Mapper (YAML)
**Решение:** YAML-маппинги в `config/mappings/`.  
**Обоснование:** Отделение бизнес-логики от кода, изменения без деплоя.

## ADR-005: Немедленный ответ 200 на вебхуки
**Решение:** Возврат 200 OK сразу после постановки в очередь.  
**Обоснование:** Исключение таймаутов со стороны OneBox.

## ADR-006: SQLite v1 → PostgreSQL v2
**Решение:** SQLite в v1, SQLAlchemy абстракция для будущей миграции.  
**Обоснование:** Нулевой overhead деплоя в v1.

## ADR-007: JSON Lines формат логирования
**Решение:** JSON Lines via `structlog`.  
**Обоснование:** Машиночитаемость, удобство агрегации (ELK/Loki).

## ADR-008: Однонаправленность внешних адаптеров в v1
**Решение:** В v1 — только outbound (Hub → сервис).  
**Обоснование:** Фокус на уведомлениях и экспорте для MVP.

## ADR-009: 1C/BAF Polling Architecture
**Решение:** Перехід від webhook-моделі до polling-архітектури (Hub самостійно опитує REST API BAF).
**Обоснование:** Більший контроль над навантаженням, можливість витягувати довідники (catalog, stores, customers) і гарантована доставка.

## ADR-010: BAF як Source of Truth
**Решение:** Ключ транзакції — `(receipt_uuid, line_no)`. Джерело правди по сумі — `line_amount`.
**Обоснование:** Дозволяє уникнути розбіжностей у фінансах між 1С та OneBox.

## ADR-011: Enriched Layer для синхронізації
**Решение:** Воркер для OneBox будується на збагаченому шарі (SQL JOIN між `fact_sales_receipt_items` та `dim_stores`, `dim_customers`, `dim_product_variants`).
**Обоснование:** Мінімізує кількість запитів під час синхронізації та забезпечує цілісність даних.

## ADR-012: Матчинг товарів через конкатенацію Артикул + Розмір
**Решение:** Артикул для OneBox генерується як `article` + суфікс розміру з `characteristic_name`.
**Обоснование:** У BAF розмір живе в характеристиці, а в OneBox це окремі товари з унікальними артикулами (напр. `429-078-QAS-XS`).

## ADR-013: Поллінг магазинів та клієнтів (Enriched Layer)
**Решение:** Написано окремий скрипт `stores_customers_sync.py` для стягування списків `/stores` та `/customers`. Запит на збагачений шар реалізовано всередині `OneBoxSyncWorker` через SQLAlchemy `JOIN` 4 таблиць (`fact_sales_receipt_items`, `dim_product_variants`, `dim_stores`, `dim_customers`).
**Обоснование:** Уникнення мікросервісного оверхеду; єдиний worker агрегує і формує JSON-payload, використовуючи гарантовано цілісні дані з БД.

## ADR-014: Попередження дублювання товарів в OneBox
**Решение:** Використання параметра `deletenotupdatedproducts: True` у payload замовлення.
**Обоснование:** OneBox API v2 за замовчуванням додає товари до існуючої угоди при збігу `externalid`. Цей прапор змушує OneBox видалити всі позиції, які не передані в поточному запиті, що дозволяє виконувати чистий "overwrite" складу чека.

## ADR-015: Мапінг магазинів та найменування угод
**Решение:** 
1. Назва угоди формується як останній сегмент номера чека (напр. `W04760`).
2. Поле `UF_CRM_1526724731` заповнюється текстовим значенням (лейблом) магазину згідно з мапінгом назв із BAF.
**Обоснование:** Вимоги бізнесу до зручності відображення даних в інтерфейсі OneBox.

## ADR-016: Проблема дублювання контактів при лінкуванні замовлень (OneBox API v2)
**Ситуація:** Виявлено, що параметр `clientid` на верхньому рівні `order/set/` ігнорується, а передача об'єкта `client` з іменем та прізвищем провокує створення дублікатів.
**Решение:** Використання структури `client: { "userid": "ID" }` у payload замовлення. Це забезпечує точне лінкування до існуючого контакту за його внутрішнім ID OneBox без створення нових записів.
**Обоснование:** Підтверджено техпідтримкою OneBox та успішними тестами лінкування.

## ADR-018: Перехід на прямий артикул характеристики (BAF characteristic_article)
**Проблема:** Поточне формування артикулу шляхом конкатенації `article` + суфікс із `characteristic_name` є ненадійним, оскільки назва характеристики може змінюватися або містити зайві символи (напр. `/`).
**Решение:** Використовувати пряме поле `characteristic_article` з BAF API як єдиний ключ для матчингу товарів у OneBox. 
**Обоснование:** Забезпечення 100% точності та стабільності синхронізації складів замовлень. Поле додано в API BAF та інтегровано в Hub (2026-03-14).

## ADR-019: Розширення схеми DWH для аналітики (P&L)
**Решение:** Впроваджено розширену структуру таблиці `dim_product_variants` та її staging-аналога. Додано поля: `characteristic_article`, `group`, `category`, `type`, `material`, `napolnenie`, `brand`, `razmer_*`, `osobennosti` (JSONB).
**Обоснование:** Необхідність формування категорійної звітності та P&L аналітики на базі PostgreSQL. Hub тепер виступає як повноцінний Data Warehouse.

## ADR-020: Дедуплікація угод OneBox за назвою чека
**Решение:** Ключ угоди в OneBox — назва чека без префікса `НФНФ-` (напр. `R30721`). Перед створенням робиться перевірка через `order/get` по `name`.
**Обоснование:** Уникнення дублювання угод при повторних синках та ручних ретраях.

## ADR-021: Дедуплікація контактів OneBox за телефоном
**Решение:** Контакти шукаються через `contact/get` по `filter.phones` (масив), телефон нормалізується до цифрового формату. Upsert контакту робиться через `contact/set` з `findbyArray: ["externalid", "phone"]`.
**Обоснование:** Уникнення дублювання клієнтів при повторному потраплянні чеків.

## ADR-022: Передача фактичної ціни чека (зі знижкою)
**Решение:** В OneBox передається ціна рядка як `line_amount / qty`, а не базовий `price`.
**Обоснование:** Сума угоди в CRM має дорівнювати фактичній оплаті в 1С/BAF.

## ADR-023: Політика обробки чеків без клієнта та повернень
**Решение:**
- повернення (`qty < 0`) -> `ignored_return`
- чеки без клієнта -> `ignored_anonymous`
- чеки з `customer_uuid`, але без даних у `dim_customers` -> залишати `pending` з `missing_customer_in_dim` для повторної спроби
**Обоснование:** Не створювати сміттєві угоди в CRM, але не губити валідні чеки з тимчасовою затримкою довідника клієнтів.

## ADR-024: Синхронізація клієнтів перед OneBox sync
**Решение:** У `baf_polling.py` перед `OneBoxSyncWorker.run_sync_batch()` примусово викликається `sync_customers()`.
**Обоснование:** Новий покупець може бути створений у BAF безпосередньо перед пробиттям чека; це зменшує race condition між чеком і довідником клієнтів.

## ADR-025: Єдиний 20-хвилинний цикл без додаткових cron/worker дублів
**Решение:** Залишено єдиний розклад кожні 20 хвилин; в межах цього ж циклу робиться polling + refresh customers + onebox sync.
**Обоснование:** Уникнення конфліктів та подвійних обробок між паралельними планувальниками.

## ADR-026: Telegram-бот з рольовим доступом (RBAC)
**Рішення:** Окремий сервіс `bot` (aiogram 3, long-polling). Ролі: `owner`, `admin`, `user`. Whitelist телефонів в `config/roles.yml` з live-reload — зміни діють без рестарту. Авторизація через Telegram contact share, телефон нормалізується до E.164. Стан авторизації зберігається в `data/bot_users.json`.
**Обґрунтування:** Власник потребує оперативного доступу до даних з телефону без доступу до серверу. Розмежування прав: продавці бачать тільки пошук клієнта, адміни — технічний моніторинг, власник — всі функції включно з касою та продажами.

## ADR-027: Inline-меню замість множинних повідомлень (⚙️ Технічний стан)
**Рішення:** Кнопка "⚙️ Технічний стан" відкриває `InlineKeyboardMarkup` з 5 кнопок. Кожна кнопка запитує тільки один розділ при натисканні (callback).
**Обґрунтування:** Початкова реалізація надсилала 4 повідомлення одночасно — перевантаження для читання. Inline-меню дає навігацію без спаму.

## ADR-028: FSM ForceReply для вводу телефону покупця
**Рішення:** Кнопка "👤 Покупець" переводить бот у стан `CustomerLookup.waiting_phone` і надсилає `ForceReply` — Telegram автоматично відкриває поле відповіді з підказкою. Наступне повідомлення обробляється як номер телефону.
**Обґрунтування:** UX: не потрібно пам'ятати формат команди `/customer 380...`. Працює і з головного меню, і з inline-меню "⚙️ Технічний стан".

## ADR-029: OneBox /product/set/ — batch одним запитом
**Рішення:** `onebox_sync.py` надсилає всі товари чека одним запитом до `/api/v2/product/set/` у масиві `productsArray`.
**Обґрунтування:** Підтверджено на реальних логах OneBox — чек з 18 товарами пройшов як єдиний запит (`status:1`, всі 18 ID повернулись в `dataArray`). Попереднє твердження про "тільки по одному" було хибним: 400 у старих чеках були спричинені некоректними артикулами, а не розміром батча.

## ADR-030: Auto-retry OneBox session token при 400
**Рішення:** `OneBoxClient._post_with_retry()` перед `raise_for_status()` перевіряє тіло 400-відповіді. Якщо містить "token in header not found" — скидає `_current_token = None` і повторює запит з новим токеном.
**Обґрунтування:** Session token OneBox протухає без явного 401. Помилка приходить як 400 + JSON body. Попередня реалізація одразу йшла в except без retry, залишаючи чеки у failed нескінченно.

## ADR-031: Periodic jobs реєструються при старті воркера (RQ 2.x)
**Рішення:** `docker-compose.yml` команда воркера: `sh -c "python -m src.scheduler.periodic && rq worker --with-scheduler default"`. `periodic.py` переписано під RQ 2.x API (`queue.enqueue_at` + `Repeat(times=99999, interval=N)`).
**Обґрунтування:** RQ 2.x не має `rq.Scheduler` (старий API). Jobs зберігаються в Redis — при флаші Redis або рестарті воркера зникають. Автоматична реєстрація при старті вирішує проблему "воркер живий але нічого не робить".
