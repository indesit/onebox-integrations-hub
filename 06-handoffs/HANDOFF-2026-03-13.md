# Handoff: OneBox Integrations Hub - 2026-03-13

## Що було зроблено
1. **Інтеграція з BAF:** Успішно піднято та перевірено 4 ендпоінти на боці 1С:
   - `receipt_lines` (продажі)
   - `product_catalog` (товари)
   - `stores` (магазини - River Mall, Блокбастер, Respublika, Corner River, Днепр, Лесі Українки)
   - `customers` (клієнти з телефонами)
2. **База Даних (PostgreSQL):** Створено базовий ETL для чеків та каталогу (`fact_sales_receipt_items`, `dim_product_variants`, staging таблиці).
3. **OneBox Sync Worker:** Імплементовано відправку угод в OneBox, додано логіку конкатенації артикулу та розміру (витягнення з `characteristic_name`).

## Поточний стан та проблеми
- Зараз API-шар над BAF повністю готовий. 
- Канонічна модель зафіксована:
  - Рядок продажу: `(receipt_uuid, line_no)`
  - Варіант товару: `(product_uuid, characteristic_uuid)`
  - Матчинг клієнта: `customer_uuid` -> `customers` -> `customer_phone`
  - Матчинг магазину: `store_uuid` -> `stores` -> `store_name`
- **Проблема:** При відправці через `order/set/` OneBox плодить дублі товарів в угоді (в угоді 127837 з'явилося 6 товарів замість 3). Треба розібратися з поведінкою OneBox API (можливо, не перезаписує, а додає при збігу `externalid`, або проблема у форматі `productinfo`).

## Наступні кроки для наступного агента/розробника
1. **Створити ETL для нових ендпоінтів:** Написати скрипти для поллінгу `/stores` та `/customers` (збереження у `stg_` та `dim_` таблиці).
2. **Enriched Layer:** Створити зведений SQL/View, який об'єднує факти продажів з усіма `dim_` таблицями (товари, магазини, клієнти).
3. **Оновити Worker:** Переписати `OneBoxSyncWorker` так, щоб він читав з Enriched Layer, брав телефон клієнта і магазин, а потім формував фінальний payload для `order/set/`.
4. **Пофіксити дублювання:** Знайти причину дублювання товарів в OneBox при оновленні чека.
